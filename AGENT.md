# AGENT.md

이 파일은 이 저장소에서 작업하는 AI 에이전트와 개발자가 따라야 할 작업 원칙을 정의한다. 목적은 LLM 기반 코딩에서 자주 발생하는 과도한 구현, 숨은 가정, 불필요한 리팩터링, 검증 없는 변경을 줄이는 것이다.

이 지침은 속도보다 정확성과 유지보수성을 우선한다. 단순하고 사소한 작업에서는 과도한 절차를 만들지 말고, 필요한 범위 안에서 판단한다.

## 1. 코딩 전에 먼저 생각한다

가정하지 말고, 혼란을 숨기지 말고, 선택지와 트레이드오프를 드러낸다.

구현 전에 다음을 확인한다.

- 요구사항의 해석이 여러 가지라면 조용히 하나를 선택하지 말고 해석 후보를 명시한다.
- 불확실한 전제는 명확히 적고, 결과에 영향을 주는 경우 질문한다.
- 더 단순한 접근이 가능하면 먼저 제안한다.
- 사용자의 요청이 목적, 구현 가능성, 유지보수성, 검증 가능성과 충돌하면 이유를 설명하고 대안을 제시한다.
- 무엇이 애매한지 알 수 없을 때는 작업을 진행하지 말고 애매한 지점을 이름 붙여 확인한다.

예: "사용자 데이터 내보내기 기능"은 전체 사용자 내보내기인지, 필터된 결과인지, API 응답인지, 파일 다운로드인지, 민감 필드를 포함하는지 먼저 확인해야 한다.

## 2. 단순성을 우선한다

오늘의 문제를 해결하는 최소 구현을 선택한다. 아직 요구되지 않은 미래 확장성을 위해 복잡도를 추가하지 않는다.

- 요청받지 않은 기능을 추가하지 않는다.
- 한 번만 쓰이는 코드에 추상화를 만들지 않는다.
- 요구되지 않은 설정 옵션, 플러그인 구조, 전략 패턴, 범용 프레임워크를 만들지 않는다.
- 실제로 발생할 수 없거나 요구사항에 없는 시나리오를 위해 방어 코드를 과하게 넣지 않는다.
- 200줄로 작성한 코드가 50줄로 명확히 해결될 수 있다면 단순한 쪽으로 다시 작성한다.

판단 기준: 숙련된 엔지니어가 "이 문제에 비해 과하다"고 볼 가능성이 있으면 단순화한다.

## 3. 필요한 부분만 외과적으로 변경한다

변경은 사용자의 요청과 직접 연결되는 줄에 한정한다. 주변 코드를 개선하고 싶더라도 요청 범위를 벗어나면 변경하지 않는다.

기존 코드를 수정할 때는 다음을 지킨다.

- 관련 없는 코드, 주석, 포맷을 "개선"하지 않는다.
- 고장 나지 않은 구조를 리팩터링하지 않는다.
- 개인적으로 선호하는 스타일보다 기존 코드 스타일을 따른다.
- 관련 없는 죽은 코드나 품질 문제를 발견하면 삭제하지 말고 보고한다.
- 내가 만든 변경 때문에 불필요해진 import, 변수, 함수만 정리한다.
- 기존에 있던 미사용 코드나 오래된 구조는 명시 요청 없이는 제거하지 않는다.

검증 기준: 변경된 모든 줄은 사용자의 요청으로 추적 가능해야 한다.

## 4. 목표 중심으로 실행하고 검증한다

"잘 되게 한다" 같은 모호한 목표를 검증 가능한 성공 기준으로 바꾼다.

작업을 시작할 때 가능한 한 다음 형태로 목표를 정의한다.

1. 재현 또는 확인: 현재 문제나 요구사항을 테스트, 로그, 실행 결과, 문서 기준으로 확인한다.
2. 최소 변경: 성공 기준을 만족하는 가장 작은 변경을 적용한다.
3. 검증: 테스트, 린트, 수동 실행, 문서 대조 중 적절한 방법으로 결과를 확인한다.
4. 회귀 확인: 변경 범위와 연결된 기존 동작이 깨지지 않았는지 확인한다.

예:

- "검증 추가"는 "잘못된 입력 테스트를 작성하고 통과시킨다"로 바꾼다.
- "버그 수정"은 "버그를 재현하는 테스트를 먼저 만들고 통과시킨다"로 바꾼다.
- "리팩터링"은 "변경 전후 테스트가 통과하고 외부 계약이 유지된다"로 바꾼다.

다단계 작업은 각 단계마다 확인 방법을 붙인다.

```text
1. [단계] -> 확인: [테스트 또는 점검 방법]
2. [단계] -> 확인: [테스트 또는 점검 방법]
3. [단계] -> 확인: [테스트 또는 점검 방법]
```

## 5. 흔한 실패 패턴을 피한다

다음 패턴은 코드 품질을 높이는 것처럼 보일 수 있지만, 요구사항보다 앞서 복잡도를 추가한다는 점에서 위험하다.

| 실패 패턴 | 문제 | 올바른 대응 |
| --- | --- | --- |
| 파일 형식, 필드, 범위를 조용히 가정 | 보안, 개인정보, 성능 요구를 놓칠 수 있음 | 가정을 명시하고 영향이 크면 질문 |
| 단순 계산에 전략 패턴과 클래스 계층 추가 | 이해와 테스트가 어려워짐 | 함수 하나로 시작하고 실제 요구가 생기면 확장 |
| 버그 수정 중 인접 코드 포맷과 스타일 변경 | 리뷰 범위가 흐려지고 회귀 위험 증가 | 버그를 고치는 줄만 변경 |
| "검토 후 개선" 같은 모호한 계획 | 성공 여부를 판단할 수 없음 | 재현, 수정, 검증 기준을 구체화 |
| 성능 개선 요청에 캐시, 비동기, 인덱스를 한꺼번에 추가 | 병목을 확인하지 않고 복잡도만 증가 | 어떤 성능 지표를 개선할지 먼저 확인 |
| 테스트 없이 바로 수정 | 실제 문제가 해결됐는지 알 수 없음 | 실패를 재현한 뒤 수정하고 통과를 확인 |

## 6. 리뷰와 문서 작업에도 같은 원칙을 적용한다

이 지침은 코드 작성뿐 아니라 분석, 설계, 리팩터링, 테스트 전략, 장애 분석, 문서 검토에도 적용한다.

- 설계 제안은 요구사항, 구현 가능성, 운영 가능성, 검증 가능성으로 평가한다.
- 문서는 읽는 사람이 다음 행동을 할 수 있도록 계약, 입력, 출력, 제약, 검증 방법을 분명히 적는다.
- 테스트 전략은 "무엇을 보장하는지"와 "무엇을 보장하지 않는지"를 구분한다.
- 장애 분석은 추측과 확인된 사실을 분리한다.
- 보고서는 긍정적 표현보다 재현 가능한 근거와 책임 있는 결론을 우선한다.

## 7. 작업 완료 기준

작업을 완료했다고 말하려면 다음 중 해당 항목을 충족해야 한다.

- 요청된 파일, 코드, 문서 변경이 실제로 반영되어 있다.
- 변경 범위가 요청과 직접 연결되어 있다.
- 가능한 검증을 수행했고, 수행하지 못한 검증은 이유를 명시한다.
- 남은 위험, 모호한 전제, 후속 작업이 있으면 숨기지 않고 보고한다.
- 최종 답변은 변경 내용과 검증 결과를 간결하게 포함한다.

좋은 작업의 신호는 불필요한 diff가 줄고, 과도한 추상화로 인한 재작성이 줄며, 실수한 뒤 수습하는 질문보다 구현 전 명확화가 늘어나는 것이다.

## 8. 작업기록과 PC 간 handoff 규칙

이 저장소는 GitHub로 관리되며, 한 PC에서 작업한 뒤 push하고 다른 PC에서 pull 받아 이어서 작업할 수 있어야 한다. 따라서 작업 상태는 대화창이나 로컬 IDE 상태에만 남기지 말고 저장소 안의 Markdown 파일에 기록한다.

작업기록의 기준 위치는 `_worklog/`이다.

```text
_worklog/
  INDEX.md
  HANDOFF.md
  checkpoints/
    YYYY-MM-DD-NNN-topic.md
```

각 파일의 역할은 다음과 같다.

- `_worklog/INDEX.md`: 모든 작업기록의 인덱스다. Codex가 작업을 재개할 때 가장 먼저 읽어야 한다.
- `_worklog/HANDOFF.md`: 다른 PC에서 pull 받은 뒤 바로 확인할 현재 상태, 다음 작업, 주의사항을 기록한다.
- `_worklog/checkpoints/`: 체크포인트별 상세 작업기록을 저장한다. 설계 판단, 변경 파일, 검증 결과, 남은 위험을 구체적으로 남긴다.

새 작업을 시작할 때는 다음 순서로 확인한다.

1. `AGENT.md`를 읽고 저장소 작업 원칙을 확인한다.
2. `_worklog/INDEX.md`를 읽고 작업기록 구조와 최신 체크포인트를 확인한다.
3. `_worklog/HANDOFF.md`를 읽고 현재 브랜치 상태, 다음 작업, 차단 사항을 확인한다.
4. 필요한 경우 최신 체크포인트 파일을 읽고 이전 판단 근거를 확인한다.

체크포인트는 다음 상황에서 반드시 추가하거나 갱신한다.

- 요구사항 해석이나 설계 방향이 확정되었을 때
- 코드, 스키마, 문서 구조처럼 이후 작업에 영향을 주는 변경을 했을 때
- 테스트, 검증, 실행 결과가 다음 작업의 전제가 될 때
- 작업을 중단하고 다른 PC나 다른 세션으로 넘겨야 할 때
- 사용자와 합의한 범위, 보류 사항, 위험이 생겼을 때

체크포인트 파일에는 최소한 다음 항목을 포함한다.

- 목적
- 현재 상태
- 변경/결정 사항
- 확인한 파일
- 검증 결과
- 남은 작업
- 다음 에이전트가 먼저 볼 것

`_worklog/HANDOFF.md`는 항상 최신 작업 상태를 짧고 명확하게 유지한다. 상세한 근거는 체크포인트 파일에 두고, handoff 파일에는 다음 작업자가 바로 이어서 실행할 수 있는 정보만 둔다.

작업 완료 전에는 다음을 확인한다.

- 새 체크포인트가 필요하면 `_worklog/checkpoints/`에 작성했다.
- `_worklog/INDEX.md`에 새 체크포인트 링크와 요약을 추가했다.
- `_worklog/HANDOFF.md`의 현재 상태와 다음 작업이 최신이다.
- `git status --short` 기준으로 의도하지 않은 변경이 섞이지 않았는지 확인했다.
